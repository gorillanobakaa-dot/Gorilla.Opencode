#001  —  2026-07-23  —  FIXED (v0.1.32)
Streaming connections were never closed → NVIDIA NIM "ResourceExhausted"
================================================================================

SYMPTOM
  During longer agent sessions the stream dies with:
    failed to process events: received error while streaming:
    {"message":"ResourceExhausted: Worker local total request limit reached (46/3..."}
  Never happens in official opencode 1.17.20 on the SAME Nemotron-3-Ultra model
  and the SAME NVIDIA NIM key. Does not happen on a trivial "yo"; shows up after
  the agent has done a bunch of tool-loop rounds.

LAYMAN
  Every AI call opens a "phone line" to NVIDIA. The official app hangs up when
  the answer is done; ours set the receiver down but left the line open. A long
  agent task opens a new line every step, so they stack up (1, 2, ... 46) until
  NVIDIA's switchboard hits its limit and refuses new calls. Fix: hang up after
  each call.

ROOT CAUSE (ours)
  internal/llm/provider/openai.go — stream(): each turn calls
    openaiStream := o.client.Chat.Completions.NewStreaming(ctx, params, ...)
  drains it with openaiStream.Next(), reads openaiStream.Err(), then returns
  (success) or `continue`s (retry) WITHOUT EVER CALLING openaiStream.Close().
  The openai-go SDK does NOT auto-close the body on drain (verified:
  ssestream.Stream.Close() -> decoder.Close(), and Next()/Err() never close it).
  NIM's endpoint is HTTP/2, so each un-closed SSE stream stays half-open and is
  counted by NIM as an ACTIVE in-flight request on that worker. An agentic
  session opens one stream per tool-loop round (agent.go:376, loop at agent.go:328)
  plus one per helper sub-agent — so they accumulate until the worker's in-flight
  cap is reached ("46/..."), and the next request is rejected with ResourceExhausted.

  Same latent leak existed in internal/llm/provider/anthropic.go (NewStreaming
  with no Close) — fixed for parity even though NIM uses the openai path.

WHY OFFICIAL IS FINE
  opencode-1.18.4 uses the Vercel AI SDK (streamText) with an AbortController and
  tears every stream down on exit — e.g. Effect.ensuring(() => abort.abort()) in
  packages/opencode/src/cli/cmd/providers.ts:342 and abort.abort() throughout
  packages/opencode/src/cli/cmd/run/stream.transport.ts. The fetch ReadableStream
  is always released, so no request is ever left in flight and nothing accumulates
  on NIM's side.

  General pattern: they route all LLM traffic through one AI-SDK layer that
  guarantees teardown; we hand-roll the SSE loop per provider and forgot the
  cleanup. Centralised lifecycle > hand-rolled per-provider lifecycle.

FIX
  Call stream.Close() immediately after stream.Err(), so it runs on ALL exits
  (success return, retry continue, error return):
    - internal/llm/provider/openai.go     (openaiStream.Close())
    - internal/llm/provider/anthropic.go  (anthropicStream.Close())
  Close() is safe if the stream never opened (guards on decoder == nil).

VERIFIED
  go build ./... clean · go vet clean · go test ./internal/llm/provider/ passes.

NOTE / RELATED (not the root cause, but amplifiers)
  - Helper sub-agents default to UNLIMITED (config/subagents.go:36), so a single
    turn can open many extra streams — each was leaking too. Consider bounding
    the default leash for free-tier NIM users. Mitigation today: /context leash +
    /tasks kill.
  - Pace-setter (config/ratelimit.go, DefaultRPM=25) spaces requests over TIME
    but does not cap simultaneous in-flight streams; it never addressed this leak.
